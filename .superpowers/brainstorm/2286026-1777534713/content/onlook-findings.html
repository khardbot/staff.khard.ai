Passo 2 · Clarificação de Arquitetura
⚠️
Onlook suporta APENAS Next.js + TailwindCSS — não SwiftUI
O subagente confirmou: o README do GitHub lista "Support non-NextJS projects" e "Support non-Tailwind projects" como funcionalidades futuras, ainda não implementadas. Hoje, edição visual → código funciona exclusivamente em projetos Next.js com Tailwind, rodando em web containers (CodeSandbox SDK).

Isso cria um ponto de decisão: o workflow de "edição visual diretamente no código" precisa ser adaptado para SwiftUI de outra forma — ou o app começa como web e migra depois.

Implicação para o projeto
Web-first vs iOS-native: o que muda no workflow
🌐 Abordagem Web-first (Next.js)
Onlook funciona nativamente nesse cenário
✅
Onlook, Pencil, Stitch, Neuform funcionam 100% sem adaptação
✅
Deploy instantâneo, sem App Store, iterações em horas
✅
Design + código no mesmo ambiente de edição visual
⚠️
Experiência mobile limitada (PWA vs app nativo)
⚠️
SwiftUI animations e gestos nativos não disponíveis
📱 Abordagem iOS-native (SwiftUI)
Onlook precisa ser substituído/adaptado
✅
Experiência nativa completa: gestos, haptics, widgets, Siri
✅
SwiftUI Animations nativas (referências dos vídeos baixados)
✅
Distribuição via App Store (monetização, credibilidade)
🔴
Onlook não suporta — precisa usar Xcode Previews como equivalente
🔴
Loop de prototipagem mais lento (compile → preview → ajusta)
Ponto de decisão
Qual caminho tomar para o app de residência?
A
Web-first → iOS depois
Começa como Progressive Web App (Next.js) usando todas as 4 ferramentas. Quando o produto valida, migra partes críticas para SwiftUI nativo.
Menor atrito inicial · Onlook funciona 100%
B
iOS-native com workflow adaptado
SwiftUI como alvo, mas usa Neuform + Stitch + Pencil para geração e Xcode Previews como substituto do Onlook. Onlook fica como referência conceitual apenas.
Produto final melhor · Loop mais lento
C
Paralelo: Web (protótipo rápido) + iOS (nativo)
Dois repos em paralelo. Web valida UX rapidamente com as 4 ferramentas. iOS implementa as telas validadas. Design system compartilhado (tokens em DESIGN.md).
Máxima velocidade · Maior esforço de manutenção
